# Lockfile Design — `rosup.lock`

## Why a lockfile

Without a lockfile, `rosup resolve` produces different results depending on:

- **Time** — rosdistro cache updates, new apt packages published
- **Machine** — different overlays installed, different Ubuntu version
- **ROS distro** — humble vs jazzy have different package sets

A lockfile pins the exact resolution so that `rosup resolve` on another
machine (or after a cache refresh) produces the same result.

## What Cargo locks vs what rosup needs to lock

| Aspect           | Cargo                      | rosup                                          |
|------------------|----------------------------|------------------------------------------------|
| Dependency graph | Exact version per crate    | Category per dep (ament/binary/source)         |
| Version pinning  | `name = "1.2.3"`           | Not meaningful (ROS has no version resolution) |
| Source pinning   | Registry hash              | Git commit SHA for source deps                 |
| Binary pinning   | N/A (everything is source) | apt package name (not version)                 |
| Reproducibility  | Exact same build           | Same resolution plan, not same binary          |

## What `rosup.lock` records

Following Cargo.lock's pattern: record **everything** in the resolution,
including workspace members and their versions.

```toml
# rosup.lock — auto-generated by rosup, do not edit
version = 1

[metadata]
ros-distro = "humble"

# Workspace members — name + version from package.xml
[[members]]
name = "autoware_core"
version = "1.5.0"
path = "src/core/autoware_core/autoware_core"

[[members]]
name = "autoware_utils"
version = "0.2.0"
path = "src/universe/autoware_utils/autoware_utils"

# Ament-installed deps — name + version from the overlay
[[deps]]
name = "rclcpp"
source = "ament"
version = "16.0.10"

# Binary deps — name + installer + system packages + version at lock time
[[deps]]
name = "eigen"
source = "binary"
installer = "apt"
packages = ["libeigen3-dev"]
version = "3.4.0-4"

# Source deps — name + git URL + branch + pinned commit SHA
# The SHA is the critical pin. Like Cargo.lock for git deps:
# `git+https://url#commit_sha`
[[deps]]
name = "nav2_core"
source = "git+https://github.com/ros-planning/navigation2.git#a1b2c3d4e5f6"
repo = "navigation2"
branch = "humble"

# Override deps — same as source but from [resolve.overrides]
[[deps]]
name = "custom_nav"
source = "git+https://github.com/myfork/navigation2.git#f6e5d4c3b2a1"
repo = "navigation2"
branch = "my-fix"

# Ignored deps — tracked so `--check` can verify nothing changed
[[deps]]
name = "catkin"
source = "ignored"
```

### What each section records

**`[[members]]`** — workspace packages (like Cargo's workspace members):
- `name` — package name from `<name>` in package.xml
- `version` — package version from `<version>` in package.xml
- `path` — relative path in the workspace

If a member is added/removed/renamed, `rosup resolve --check` detects the
drift.

**`[[deps]]` with `source = "ament"`:**
- `version` — read from the installed package's `package.xml` in the
  overlay (e.g. `/opt/ros/humble/share/rclcpp/package.xml`).
- **Warn if installed version is older than locked.** This means the
  developer who locked had a newer version — the current machine may
  behave differently. Does not fail; the user may be on an older overlay
  intentionally (e.g. Ubuntu LTS).

**`[[deps]]` with `source = "binary"`:**
- `installer` + `packages` — the rosdep key → system package mapping.
- `version` — the installed version at lock time (e.g. from `dpkg -s`).
- **Warn if installed version is older than locked.** Same rationale as
  ament: the developer who locked may have had a newer package with fixes
  or API additions that this machine lacks.

**`[[deps]]` with `source = "git+<url>#<sha>"`:**
- The commit SHA is the **critical pin** — without it, branch HEAD moves
- Format matches Cargo.lock: `git+<url>#<sha>`
- `repo` and `branch` for human readability

**`[[deps]]` with `source = "ignored"`:**
- Tracks suppressed deps so `--check` knows they were intentional

### How the SHA is captured

When `rosup resolve` clones a source dep:
1. Clone/fetch into `~/.rosup/src/<repo>.git`
2. Create worktree at the target branch
3. Read `git rev-parse HEAD` → commit SHA
4. Record SHA in `rosup.lock`

## CLI workflow

```bash
# First resolve — creates rosup.lock
rosup resolve
# Lockfile generated: rosup.lock

# Subsequent resolves — uses locked versions
rosup resolve
# Using locked resolution (rosup.lock)

# Update lockfile (re-resolve from scratch)
rosup resolve --update
# Lockfile updated: rosup.lock

# Check if lockfile is up to date (CI)
rosup resolve --check
# exit 0 if lock matches current resolution, exit 1 if stale
```

## When the lockfile is used

| Command                   | Uses lockfile?         | Updates lockfile? |
|---------------------------|------------------------|-------------------|
| `rosup resolve`           | Yes (prefer locked)    | Yes (if missing)  |
| `rosup resolve --update`  | No (re-resolve)        | Yes               |
| `rosup resolve --check`   | Yes (compare)          | No                |
| `rosup build`             | No (only checks)       | No                |
| `rosup resolve --dry-run` | Yes (show locked plan) | No                |

## Should `rosup.lock` be committed?

Like Cargo:
- **Applications/workspaces: yes** — reproducible builds across CI and team
- **Libraries (single packages): maybe** — depends on whether the package
  is used standalone or as part of a larger workspace

## Lockfile vs existing resolver

The lockfile is a **cache of the resolution plan**. When present:

1. Read `rosup.lock`
2. For each locked dep, check if the lock is still valid:
   - `ament` — still installed? OK
   - `binary` — package still available? OK
   - `source` — repo accessible? Use locked SHA
3. If any dep is missing from the lock (new dep added to package.xml),
   resolve it fresh and append to the lock
4. If `--update`, ignore the lock and re-resolve everything

## Implementation plan

1. Define `LockEntry` struct and serialization format (TOML)
2. Add `rosup.lock` read/write to the resolver
3. When resolving source deps, record the commit SHA after clone
4. Add `--update` and `--check` flags to `rosup resolve`
5. Add `rosup.lock` to the resolver's plan flow

## Version comparison behavior

When `rosup.lock` exists, `rosup resolve` compares installed versions
against locked versions:

| Situation | Behavior |
|-----------|----------|
| Installed version == locked | OK, silent |
| Installed version > locked | OK, silent (newer is fine) |
| Installed version < locked | **Warning** — older than locked |
| Package not installed | Error — must resolve |
| Package not in lockfile | Resolve fresh, append to lock |

Example output:
```
warning: rclcpp 16.0.8 is older than locked 16.0.10
warning: libeigen3-dev 3.4.0-2 is older than locked 3.4.0-4
hint: run `rosup resolve --update` to update the lockfile
```

Warnings are advisory — `rosup resolve` still succeeds. The build may
or may not work depending on whether the version difference matters.

- **Lock freshness** — How to detect that `rosup.lock` is stale? Check:
  1. All workspace members in the lock still exist (and versions match)
  2. All deps in the lock are still in the dep set (no new deps added)
  3. Source dep SHAs match what's checked out in `.rosup/src/`

- **Docker/CI reproducibility** — The lockfile's main value is in CI where
  you want `rosup resolve` to produce identical results to the developer's
  machine. Source SHA pinning achieves this. Binary deps are inherently
  non-reproducible across different base images unless you also pin the
  Docker base image.
